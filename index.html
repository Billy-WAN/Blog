SpringBoot概述

 

SpringBoot诞生的意义是简化Spring应用的开发，是整个Spring技术栈的大整合，J2EE开发的一站式解决方案。

特点是约定大于配置，去繁从简，迅捷开发部署。

 

优点：

快速创建独立运行的Spring项目以及与主流框架集成；
使用嵌入式的Servlet容器（Tomcat服务器），应用无需打成WAR包；
Starters自动依赖管理与版本控制；
大量的自动配置，简化开发，同时也可以修改默认值；
基本无需配置XML，采用注解配置，无代码生成，开箱即用；
准生产环境的运行时应用监控；
与云计算的天然集成。

微服务

2014年Martin Fowler正式提出了“微服务”的概念。将功能分解到离散的各个服务当中，从而降低系统的耦合性，并提供更加灵活的服务支持。

微服务是SOA架构下的最终产物，该架构的设计目标是为了肢解业务，使得服务能够独立运行。

微服务设计原则：1、各司其职 2、服务高可用和可扩展性。

微服务应用的一个最大的优点是，它们往往比传统的应用程序更有效地利用计算资源。这是因为它们通过扩展组件来处理功能瓶颈问题。这样一来，开发人员只需要为额外的组件部署计算资源，而不需要部署一个完整的应用程序的全新迭代。

微服务应用程序的另一个好处是，它们更快且更容易更新。当开发者对一个传统的单体应用程序进行变更时，他们必须做详细的QA测试，以确保变更不会影响其他特性或功能。但有了微服务，开发者可以更新应用程序的单个组件，而不会影响其他的部分。测试微服务应用程序仍然是必需的，但它更容易识别和隔离问题，从而加快开发速度并支持DevOps和持续应用程序开发。

第三个好处是，微服务架构有助于新兴的云服务，如事件驱动计算。类似AWS Lambda这样的功能让开发人员能够编写代码处于休眠状态，直到应用程序事件触发。事件处理时才需要使用计算资源，而企业只需要为每次事件，而不是固定数目的计算实例支付。

配置文件优先级

SpringBoot启动时会扫描多个位置的application.properties或者application.yml文件作为默认配置文件，所有位置的文件都会被加载，但是高优先级的配置会覆盖低优先级的配置中的相同内容，低优先级配置中的不同内容则会和高优先级配置形成互补。

优先级由高到低的顺序为：

-file:./config/
-file:./  工程根目录
-classpath:/config/
-classpath:/  类路径根目录

可以通过过配置spring.config.lacation来改变这个默认配置。

日志

 

日志就像飞机上的黑匣子，记录了现场，可以让我们复盘使用。

网络的日志，web服务器的日志，应用服务器的日志，数据库日志等等。都需要在必要的时候进行查看。如果不知道出现了什么问题，又何来的解决方法呢?

 

日志框架

包括日志门面（日志的抽象层）和日志实现两部分。

日志门面以 SLF4J 为代表，日志实现以 logback 为代表。SpringBoot选用的就是这两者。

 

 

SLF4J

 

日志记录方法的调用应该调用日志抽象层的方法。

日志的实现框架都有自己的配置文件，需要根据具体框架来编写。

 

SLF4J 框架针对项目中应用不同日志框架的统一问题也提供了转换方案：使用中间包来替换原框架。

 

 

官网简单示例：

xxxxxxxxxx
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
​
public class HelloWorld {
  public static void main(String[] args) {
    Logger logger = LoggerFactory.getLogger(HelloWorld.class);
    logger.info("Hello World");
  }
}

 

总结：
1、SpringBoot底层使用self4j+logback的方式进行日志记录；
2、SpringBoot通过中间包把其他日志工具替换成了self4j，即自动适配所有日志；
3、如果要引入其他框架，一定要把这个框架的默认日志依赖移除。

SpringBoot默认完成日志的配置

设置日志输出级别：

由低到高依次是

日志输出级别可以在application.properties配置文件中编写指定，日志会在高于等于指定值级别的级别范围中输出日志，SpringBoot默认 info 级别

​
#更改root级别就是修改SpringBoot默认的全局级别
logging.level.root=INFO

 

设置日志输出文件或者目录：

xxxxxxxxxx
logging.file.name=springboot.log
或者
logging.file.path=/springLog
​
#path更常用，指定目录自动生成默认名称的日志文件

 

设置日志的输出格式：

xxxxxxxxxx
#在控制台输出的日志格式
logging.pattern.console=%d{yyyy/MM/dd-HH:mm:ss} [%thread] %-5level %logger- %msg%n
​
#在日志文件中输出的格式
logging.pattern.file=%d{yyyy/MM/dd-HH:mm} [%thread] %-5level %logger- %msg%n
​
​
#上述编码符号含义如下：
%d{HH:mm:ss.SSS}——日志输出时间
​
%thread——输出日志的进程名字，这在Web应用以及异步任务处理中很有用
​
%-5level——日志级别，并且使用5个字符靠左对齐
​
%logger- ——日志输出者的名字
​
%msg——日志消息
​
%n——平台的换行符
​

 

指定配置

全局配置文件（application.properties）中可以指定一部分日志设置，但是完整的配置还需要通过类目录下特定的配置文件来指定，如果存在指定配置文件，则SpringBoot的默认配置信息则不生效。

对于logback框架来说，指定配置文件名可以是：logback.xml，logback-spring.xml，logback.groovy，logback-spring.groovy。

区别在于：
logback.xml——直接被日志框架识别；
logback-spring.xml——日志框架不直接加载配置，由SpringBoot来解析配置文件，可以使用SpringBoot的profile高级功能。

5、ConfigurableWebBindingInitializer

 

SpringBoot自动配置很多组件：

1、部分组件只能生效一个：SpringBoot会判断容器中是否有用户自定义的组件存在，如果有则使用，如果不存在则自动配置默认的组件进容器；

2、部分组件可以多个生效：SpringBoot会将用户配置的组件和自身默认配置的组件组合到一起。

 

 

扩展SpringMVC

编写配置类（@Configuration），继承自 WebMvcConfigurerAdapter，不能标注 @EnableWebMvc。容器中所有的WebMvcConfigurer会同时起作用，这样既保留了自动配置，还可以加入自定义扩展的配置。

 

 

@EnableWebMvc：使用该注解意味着禁用 SpringBoot 对 SpringMVC 的所有自动配置，也就是全面接管SpringMVC 配置的控制权。

禁用原理：@EnableWebMvc注解导入了DelegatingWebMvcConfiguration.class，这个类继承自WebMvcConfigurationSupport，而自动配置类WebMvcAutoConfiguration有一个条件判断注解@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)就失效了，所以自动配置不运行。

集成Redis

1、添加依赖座标：

xxxxxxxxxx
<!--springboot集成redis的起步依赖-->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

2、在springboot核心配置文件中添加redis的配置：

xxxxxxxxxx
spring.redis.host=127.0.0.1
spring.redis.port=6379
spring.redis.password=password

3、业务层利用 RedisTemplate 对象实现对redis数据的操作。

xxxxxxxxxx
示例：
@Service
public class StudentServiceImpl implements StudentService {
​
    @Autowired
    private RedisTemplate<Object,Object> redisTemplate;
    
    @Override
    public void put(String key, String value) {
        redisTemplate.opsForValue().set(key,value);
    }
    @Override
    public Object get(String key) {
        Object o = redisTemplate.opsForValue().get(key);
        return o;
    }
}

i18n国际化

i18n，来源于英文单词 internationalization的首末字符i和n，18为中间的字符数，是“国际化”的简称。在不修改内部代码的情况下，能根据不同语言及地区显示相应的界面。 相关联的还有 L10n（localization）。

 

在resources目录下新建 i18n目录，然后新建所需的属性文件以及不同的国际化语言属性文件。

 

模板引擎

在前后端一体的时代，服务器端要负责渲染html页面向用户端发送，所以需要模板引擎这样的应用进行开发，常见的如JSP、Velocity、Freemarker、Thymeleaf等（SpringBoot集成了Thymeleaf，语法简单生产力强、功能强大）；

目前正在前后端分离的潮流中，页面由前端渲染，服务器端以 json 形式提供数据，前后端交互只涉及到纯数据，后端不参与渲染模板，模板引擎类应用的被需要程度日渐式微。
